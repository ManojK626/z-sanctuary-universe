# ZUNO-4ROOT communication spine

## 1. Purpose

Define how ecosystem **signals** move from users, modules, and AI assistants into **observable**, **governed**, **human-supervised** handling. This doc is **doctrine**: it does not add backend execution or silent automation.

> ZUNO-4ROOT improves ecosystem awareness but does not grant autonomous authority. Serious actions require lawful human review.

## 2. Four-root communication model

| Root | Role |
| ------------ | ------------------------------------------------------------------------------------------- |
| **AMK-Goku** | Human creator — vision, consent, lawful accountability |
| **SKK** | Protective structure — discipline, gates, structural integrity |
| **RKPK** | Care and continuity — compassion, pacing, sustainable rhythm |
| **Zuno** | Reflection and audit — clarity, coverage hints, evidence surfacing (not autonomous command) |

Symbolic layout (conceptual, not theological):

```text
          ZUNO
           ▲
           │
SKK ◄── AMK-GOKU ──► RKPK
           │
           ▼
      LIVING ROOT
```

## 3. AI-to-AI signal flow

Internal bot-to-bot or agent suggestions **must not** bypass human gates for serious actions. Preferred posture: **observe → classify → surface to dashboard / report → human decision**.

## 4. User input flow

Operator or creator input enters through normal product surfaces (dashboard, IDE, docs). **Classification** tags intent (question, change request, deploy ask, safety concern). **Serious** intents route to review before repo or infra mutation.

## 5. System health flow

Health signals (verify scripts, registry audits, structure checks) feed **read-only** dashboard representation where wired. Where metrics are not implemented, the dashboard shows **doctrine placeholders** — never fabricated live numbers.

## 6. 14DRP governance flow

Fourteen DRP protocols anchor **non-harm** and compassion-aligned discipline. Signals pass a **14DRP check** as a **human-operated** ritual (policies, checklists, reviews) — not an automated moral oracle. See [14DR_ALIGNMENT.md](14DR_ALIGNMENT.md).

## 7. Audit and evidence flow

Outputs land in agreed artefacts: reports under `data/reports/`, Markdown receipts, registry JSON. **Evidence-first:** claims follow files and commands that exist in the hub.

## 8. STOP_GATE escalation flow

When a signal hits policy conflict, missing consent, or sacred-move territory: **STOP_GATE** — halt automation suggestions, require explicit human approval path, document reason. No silent retry loops.

## 9. Human review flow

**Serious actions** (merge, deploy, payment, legal/safety copy, public launch, external connectors, cross-product coupling, bulk data deletion/export) require **lawful human oversight** per hub gates (Z-MAOS, execution enforcer, AMK consent doctrine).

## 10. Future expansion rules

Any new module SHOULD declare (in registry + docs):

```text
module_id
purpose
input_signals
output_signals
risk_level
14DRP_alignment
human_review_required
dashboard_visibility
audit_log_path
```

