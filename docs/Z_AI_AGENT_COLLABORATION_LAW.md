# Z-Sanctuary — AI Agent Collaboration Law

**Status:** Universal governance (docs only). **Scope:** ChatGPT, Cursor AI, GitHub Copilot, and future AI agents working in or on behalf of the Z-Sanctuary Universe ecosystem.

**Authority order:** [Z-HIERARCHY-CHIEF-AND-OBSERVER-VIEW.md](Z-HIERARCHY-CHIEF-AND-OBSERVER-VIEW.md) → this law → agent-specific rules (Turtle Mode, GitHub gate, 14 DRP) → phase receipts.

**Related:** [Z_SWARM_14DRP_UNIVERSAL_AGENT_LAW.md](Z_SWARM_14DRP_UNIVERSAL_AGENT_LAW.md), [.cursor/rules/z-turtle-mode-cursor-agents.mdc](../.cursor/rules/z-turtle-mode-cursor-agents.mdc), [Z-GITHUB-SANCTUARY-GATE.md](Z-GITHUB-SANCTUARY-GATE.md), [AGENTS.md](../AGENTS.md).

---

## Standing law

```text
Human-authored governance > autogenerated convenience.
AI assists. Humans decide.
Evidence before expansion.
Safety before scale.
Mock readiness ≠ production approval.
```

---

## 1. Human authority

| Rule                | Meaning                                                                                                                                                      |
| ------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------ |
| Final authority     | Humans (AMK-Goku and designated operators) remain the **final** authority for merge, deploy, pricing, legal copy, and sacred moves.                          |
| Advisory AI         | AI suggestions are **advisory** — plans, diffs, checklists, and receipts — not binding orders.                                                               |
| No deploy authority | **No** AI agent may claim deployment authority, production approval, or override of `manual_release` / execution enforcer gates.                             |
| Sacred moves        | Merge to `main`, deploy, payments, public launch, bulk user-data actions, and cross-repo runtime bridges require **explicit human consent** per hub charter. |

---

## 2. Turtle Mode

| Principle     | Practice                                                                                                          |
| ------------- | ----------------------------------------------------------------------------------------------------------------- |
| Pace          | Slow, **evidence-first** progress; smallest safe step.                                                            |
| Reversibility | Prefer small, **reversible** changes; document rollback before expanding scope.                                   |
| Receipts      | Phase work ships with green receipts where the lane requires them; no “done” without verify output when mandated. |
| Branches      | Use `cursor/zsanctuary/*` (Cursor); never edit `main` directly for agent work.                                    |
| Amber → green | Do not flip readiness to green without restore drill, sign-off, and verification logs where policy requires.      |

---

## 3. PR discipline

| Requirement               | Detail                                                                                                                                          |
| ------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------- |
| Narrow scope              | One domain per PR (docs, UI, scripts, registry — not several).                                                                                  |
| No unrelated churn        | No drive-by table reformats, encoding “fixes” outside scope, or whole-file prettier on unrelated paths.                                         |
| No report pollution       | Do **not** commit `data/reports/*`, `data/system-status.json`, or `data/z_bridge/logs.json` unless the PR **explicitly** owns that report lane. |
| No hidden generated files | No accidental artifacts, `.env`, caches, or local-only outputs in the tree.                                                                     |
| Reviewable diffs          | Keep branches short-lived; reviewers must see **why** each line changed.                                                                        |

---

## 4. Doctrine before runtime

| Order | Deliverable                                                                        |
| ----- | ---------------------------------------------------------------------------------- |
| First | Docs, receipts, governance, registry entries (when chartered).                     |
| Then  | Read-only dashboards and **mock** UI.                                              |
| Last  | Runtime features, APIs, networking, deploy — only with phase receipt + human gate. |

**Mock readiness does not equal production approval.** A static cockpit, simulation, or blue receipt is **not** permission to bind production, track users, or expose NAS/edge.

## 5. Privacy + 14 DRP

Align with [Z_SWARM_14DRP_UNIVERSAL_AGENT_LAW.md](Z_SWARM_14DRP_UNIVERSAL_AGENT_LAW.md) and hub vault policy.

| Forbidden                                                     | Required                                                     |

| Hidden profiling or cross-session surveillance                | Data minimization; local-first where phases allow            |

## 6. AI agent separation

| Agent                                      | Role                                                                                                 | Does not                                                                    |
| ------------------------------------------ | ---------------------------------------------------------------------------------------------------- | --------------------------------------------------------------------------- |
| **ChatGPT** (and similar chat strategists) | Strategic reasoning, governance guidance, option merge hints, doctrine drafting **for human review** | Implement repo changes without human transfer; claim merge/deploy authority |
| **Cursor AI**                              | Implementation assistant: edits, verify commands, PR-sized branches, docs/mocks per Turtle Mode      | Auto-merge, auto-deploy, silent background mutation, bypass approvals       |
| **GitHub Copilot**                         | Scaffolding, completions, **helper** suggestions in-editor                                           | Own the architecture; override hub doctrine; wide refactors without review  |
| **Human**                                  | Final architect, operator, merge authority, charter issuer                                           | Delegate sacred moves to automation                                         |

Future agents (cloud workers, SDK agents, mini-bots) must register against this table before receiving write or network scope.

---

## 7. Forbidden behaviors

All agents **must not**:

- Claim **fake AGI** or supernatural authority over the ecosystem.
- **Auto-deploy**, auto-merge to `main`, or force-push without explicit human request.
- Perform **silent mutations** (files, config, registry, secrets) outside stated scope.
- Make **hidden network calls** (undeclared APIs, telemetry, package installs) without charter.
- Invent **metrics**, green status, or “production ready” language without receipt proof.
- Use **false production language** (“live”, “deployed”, “authorized”) for mocks or docs-only phases.
- **Overwrite human doctrine** — hub docs and AMK-gated policy outrank model output; thin pointers only in rules files.

## 8. Stash / branch hygiene

| Practice                                  | Why                                                                                          |
| ----------------------------------------- | -------------------------------------------------------------------------------------------- |
| Inspect before `stash drop` / `stash pop` | Avoid losing operator WIP or applying wrong context                                          |
| Rebase cleanly onto current `main`        | Prevent duplicate commits and mixed-phase stacks                                             |
| Avoid mixed-purpose branches              | e.g. do not combine AetherNav + legal UX + reports in one PR                                 |
| Restore accidental churn                  | If an editor widens tables across unrelated files, `git restore` before commit               |
| One lane per branch name                  | Branch name should match the phase doc (e.g. `cursor/zsanctuary/ai-agent-collaboration-law`) |

---

## 9. Collaboration flow (observe → human decides)

```text
observe → verify → suggest → human decides
```

1. **Observe** — read Hierarchy Chief, phase receipt, and relevant registry/report (read-only).
2. **Verify** — run mandated commands (`verify:md`, lane-specific verify); paste results.
3. **Suggest** — smallest safe diff, file list, rollback plan.
4. **Human decides** — merge, defer, or redirect; AI does not execute sacred moves.

## 10. Rollback and verification

### Verification (this document)

```bash
npm run verify:md
```

### Acceptance (docs-only change)

- This file exists at `docs/Z_AI_AGENT_COLLABORATION_LAW.md`.
- Sections 1–9 and standing law are present.
- **No** runtime, dashboard, registry, or `data/reports/*` changes in the same commit unless a separate chartered PR owns them.

### Rollback

```bash
git revert <commit-sha>
npm run verify:md
```

Or drop the branch before merge if the PR is not yet integrated.
