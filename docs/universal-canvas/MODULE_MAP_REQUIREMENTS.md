# Z-UCCR — Module Map Requirements

**Purpose:** Describe how a **living constellation map** represents Z-Sanctuary modules as **spatial memory**, not Org Chart surveillance.

---

## 1. Map philosophy

- Modules appear as **nodes** (clusters by spine: Oversight / Data / Build / Cognitive parallels to OMNI organism).
- Edges indicate **sanctioned relationships** only (imports from docs/manifest)—never inferred illicit data flows.
- **Read-only first:** pan/zoom/select opens doc or receipt; no dragging that changes filesystem until Z-UCCR-3+ launcher discipline exists.

---

## 2. Data sources (read path)

| Source | Map use |
| -------------------------------------- | --------------------------------------------------------- |
| `data/Z_module_registry.json` (+ sync) | Node list, statuses |
| Master Register / checklist | Coverage gaps |
| `z_pc_root_projects.json` | Sibling repos as outer orbit (names only unless expanded) |

---

## 3. Allowed visuals

- Node states: stub / staged / wired / gated.
- Hover: short description + link to blueprint.
- Halo color = verify posture aggregate (amber allowed from real reports).

---

## 4. Forbidden visuals

- “Security score” for modules without methodological backing.
- Edges implying live network intrusion paths.
- Hiding gated modules entirely without STOP indicator when policy demands visibility.

---

## 5. Phase

- **Z-UCCR-1:** Static constellation screenshot + legend in wireframe repo or doc.
- **Z-UCCR-2:** Generated graph from sanitized JSON snapshots (offline script later; not required by doctrine).
